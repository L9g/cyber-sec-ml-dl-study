# Model Selection Decision — Tree Boosting Baseline

> 前置报告：本文件记录项目开始前对主模型的选择理由。它不是事后为结果找理由，而是把工程取舍、
> 个人熟悉度、评估目标和已知风险写清楚，避免把模型选择伪装成“理论最优”。

## 0. 项目目标先于模型选择

本项目的目标不是刷新 NIDS benchmark 分数，而是做一个可复现的评估审计 demo：

1. 量化随机切分、身份特征、时间泄漏等方法学问题如何制造虚高结果。
2. 用 temporal split、特征卫生和 LODO 暴露单数据集高分与跨数据集泛化之间的断层。
3. 把指标放回安全运营语境：base rate、FPR、recall、PR-AUC、告警量和模型风险。

因此主模型需要满足四个条件：

| 要求 | 原因 |
|---|---|
| 强 tabular baseline | NetFlow 是结构化表格特征，树提升模型是合理起点 |
| 默认参数可运行 | 项目卖点是评估方法，不是大规模调参竞赛 |
| 大数据可承受 | v3 的 ToN-IoT/CSE-CIC 有千万级流，模型不能成为主要瓶颈 |
| 结果容易解释和复查 | portfolio 读者需要能看懂失败来自数据/评估，而不是黑盒技巧 |

## 1. 候选模型范围

候选限定为主流 gradient boosting tree 系列：

- **LightGBM**
- **XGBoost**
- **CatBoost**

不把深度学习作为主模型，原因是：本项目的核心论点是 benchmark 评估不诚实，而不是“更大模型更好”。
深度模型可作为后续鲁棒性扫描或延伸方向，但不应抢主叙事。

## 2. 决策矩阵

评分：高 / 中 / 低，按本项目语境判断，不代表通用排名。

| 维度 | LightGBM | XGBoost | CatBoost |
|---|---|---|---|
| 个人熟悉度 | **高**：我对 API、参数和常见 failure mode 熟悉 | **高**：同样熟悉，适合作为替代验证 | 中：了解但不是最熟 |
| 大规模 tabular 训练速度 | **高**：histogram + leaf-wise，千万级抽样实验更省时 | 中-高：强但通常更慢，内存压力可能更高 | 中：类别处理强，但本项目没有保留高价值类别特征 |
| 默认参数 baseline 质量 | **高**：默认参数已经足够做评估审计 | 高：默认也强，常见工业 baseline | 高：默认鲁棒，尤其有类别变量时 |
| 对 NetFlow 数值特征的适配 | **高**：本项目主要是数值 flow 统计 | 高 | 中-高 |
| 对类别特征的优势 | 中：需自行编码 IP/端口等 object 列 | 中：同样需编码 | **高**：CatBoost 的核心优势 |
| 与项目特征卫生的关系 | **好**：IP/端口本应删除，不需要模型擅长利用类别 ID | 好 | 一般：CatBoost 的类别优势可能鼓励保留本应删除的身份特征 |
| 运行成本 / 依赖复杂度 | **低成本**：已在 requirements 中稳定使用 | 中：若加入需新增依赖和复跑结果 | 中：新增依赖，叙事解释成本更高 |
| 解释风险 | 中：树模型仍可能学习 benchmark 指纹 | 中：同类风险 | 中-高：类别处理机制容易把讨论带到编码细节 |
| 面试可解释性 | **高**：我能清楚解释为何用它、为何它也会失败 | 高 | 中：需要额外解释为何不用其类别能力 |
| 作为主模型结论稳定性 | **高**：配合 stump/LogReg/MLP 扫描，可证明不是 LightGBM 特例 | 高：可作为复核模型 | 中：适合作扩展，不适合作主叙事 |

## 3. 决策

**主模型选择 LightGBM。**

理由分三层：

1. **工程理由**：LightGBM 在大规模 tabular 数据上快，默认参数足够强，适合 2M-3M 行级别的 LODO
   反复实验。
2. **项目理由**：本项目要证明评估口径和数据分布才是主问题。LightGBM 足够强，若它在分布内满分、
   跨数据集崩塌，这个失败更有说服力。
3. **个人理由**：我对 LightGBM 和 XGBoost 最熟悉，能更快定位问题、解释结果、排除用法错误。
   这是真实工程选择的一部分，不应事后包装成纯理论判断。

XGBoost 是最自然的备选：若需要增强说服力，可用同一特征集和同一 split 复跑一个小规模 XGBoost
对照，验证“跨数据集崩塌不是 LightGBM 独有”。但在 MVP 阶段，它不改变主结论，只增加运行成本。

CatBoost 暂不作为主模型。它的类别特征能力在许多表格任务中有价值，但本项目的关键类别/身份特征
（IP、端口）恰恰是要删除或隔离的环境特征。引入 CatBoost 容易把叙事带偏到“如何更好利用类别 ID”，
而不是“为什么这些 ID 不该作为可迁移检测证据”。

## 4. 不是模型竞赛

这个项目不追求以下叙事：

- “LightGBM 比 XGBoost/CatBoost 更适合 NIDS。”
- “调参后可以把跨数据集结果救回来。”
- “最强模型就是最可信模型。”

更准确的叙事是：

> 在 NetFlow v3 这类合成 benchmark 上，强表格模型能轻易获得分布内高分；但一旦切到另一个数据集，
> PR-AUC 和 recall 会显著退化。问题主要来自数据集指纹、评估口径和部署分布差异，而不是 LightGBM
> 这个具体算法。

因此，模型选择的角色是提供一个可信、强、可复现的 baseline，而不是成为项目本身的主角。

## 5. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| LightGBM 太强，可能记住 benchmark 指纹 | 分布内满分被误读为检测能力 | 用 temporal split、去 IP/端口、LODO 和多模型扫描拆解 |
| 结果被质疑为 LightGBM 特例 | 削弱跨数据集崩塌论点 | 已加入 SingleFeature stump / LogReg / MLP / LightGBM 扫描；可后续加 XGBoost |
| 默认参数未调优 | 可能被认为实验不充分 | 明确本项目目标是评估审计；调参不是主变量 |
| 类别特征编码影响 IP 消融 | IP-only LODO 数字可能含编码噪声 | 把 IP 消融解释为“身份特征不可迁移”的机制实验，不把跨域数值当真实性能 |
| 只用合成公开数据 | 无法证明真实 SOC 部署效果 | 在 findings 中写明局限；后续可引真实流量或 production-like telemetry 做 drift/alert-budget 分析 |

## 6. 后续模型扩展优先级

如果要把 portfolio 从 MVP 升级到更强的工程展示，建议按以下顺序扩展：

| 优先级 | 模型 | 目的 |
|---|---|---|
| P0 | SingleFeature stump | 证明 UNSW 上单特征也能接近满分，支撑“benchmark 平凡可分” |
| P0 | LogReg | 线性基线，验证不是复杂模型才会高分/崩塌 |
| P0 | LightGBM | 主模型，强 tabular baseline |
| P1 | XGBoost | 作为 LightGBM 的同族复核，增强工业可信度 |
| P1 | RandomForest / ExtraTrees | 验证 bagging tree 也无法解决跨域问题 |
| P2 | CatBoost | 仅在明确处理合法类别特征时加入，不用于鼓励保留 IP/端口 |
| P2 | MLP / small DL | 展示“加模型容量不等于泛化”，但避免喧宾夺主 |

## 7. 最终记录

本项目当前选择：

- 主实验：**LightGBM**
- 鲁棒性扫描：SingleFeature stump / LogReg / MLP / LightGBM
- 暂不加入主线：XGBoost / CatBoost

选择 LightGBM 的理由包括个人熟悉度、工程效率和强 baseline 价值。这个选择不声称 LightGBM 在理论上
优于 XGBoost 或 CatBoost；它只是当前项目目标下最务实、最容易复现和解释的主模型。
