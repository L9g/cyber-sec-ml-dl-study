# 05 · Ithuriel — 安全 + 合规 Agent（红队 / 保证 anchor）

> 名字取自弥尔顿《失乐园》里那杆一触即现原形的矛 Ithuriel。项目做的正是同一件事：用探测让 AI 系统与传统系统里隐藏的失败现形。

## 这是什么

一个以保证（assurance）与对抗性故障发现为主线的安全合规 Agent。它把安全标准蒸馏成机器可读的 ontology，驱动现成工具去测一个目标系统，再把工具的原始输出凝成一份可审计、可复现、带明确边界的结论。平台本身只是底座，价值在结论层。

想快速理解设计原理与代码组织，读 `docs/DESIGN.md`。

两条产品线共享同一内核（ontology、Region Profile、RoE、插件接口、证据模型、报告）：

- **Pack A，网络与 IT 系统安全**：以 Cyber Essentials 就绪度审计为主体，附一层授权红队切片。
- **Pack B，AI Agent 风险检测**：对陌生 AI Agent 冷启动地探测其隐私、安全、合规、越界失败模式。当前切片跑在 AgentDojo 的 mock 场景上；真实部署的 OSS agent（例如 RAGFlow）是后期高保真度档的靶标，不是本项目本身。

## 核心原则：两层模型（借底座，建差异化）

- **Base 等于借**：执行机制、进程间通信、调度、以及所有扫描与探测工具，能借就不自建。如今已借入 AgentDojo（AI 注入探测）、`transformers` 生态的注入检测器、系统自带的 `ufw`（防火墙配置读取）、nmap（端口探测）；未来还会借 nuclei、garak、PyRIT，乃至一个 Rust 可信核心。它们只是可信的被审计对象与执行手段。要精确一点：借的是执行**机制**，而把执行**接缝**建成强制执行点（executor 作为 PEP，负责 RoE 授权、两阶段校验、命令白名单）是差异化层自己的事，不算借。
- **Differentiator 等于建**：只有两样自己投入。第一是把标准蒸馏成 ontology。第二是证据、Finding 与保证层，包括四态 Finding、AI 探针的运行统计 `ai_run_record`、内容寻址的证据清单 `EvidenceManifest`、跨控制的覆盖率 rollup `CoverageLedger`，以及把裁定溯源升成结构、据此产出带边界保证结论的 Claim 层（typed `verdict_provenance` 与多维 `confidence_basis`）。

Base 层遵循 Unix 式的极简与可组合，差异化层刻意重投入。插件是对现成工具的薄适配器，不是手搓专用扫描器。缺工具的控制不硬凑，如实留成覆盖缺口。

## 状态

差异化层已经跨四种裁定形状、四个控制验证过跨域通用性，四条端到端切片全部合入主干，测试全绿：

- AI 注入探测 `AI-AGENT-PI-01`（非确定 AI，跑在 AgentDojo mock 上）
- 防火墙默认拒绝策略 `CE-UK-FW-03`（确定性配置读取）
- 端口探测 `CE-UK-FW-01`（主动探测，带执行与授权机器）
- 安全更新治理 `CE-UK-SU-03`（人工复核，声明式证据）

四条切片横跨三个域、三个严重度档、两种裁定模式，由同一套 Finding、Evidence、Assurance、CoverageLedger 承载，跨这四种形状零 schema 改动。

在此之上又落了差异化层「标准到保证结论」的上半边：一个把裁定溯源升成结构的 typed `verdict_provenance`，加一个纯函数 Claim 层，产出多维的 `confidence_basis`（warrant）而不是一个会把覆盖率、证据质量、目标保真度压成一个数字的单一置信度档。这一步据真实摩擦而定，做法与纪律记在 ADR-0016。这条 Claim 层也已用真实模型跑通验证：一次多目标真跑里，系统在无正对照时正确地拒绝把裸跑零成功率当成「安全」，在两臂溯源指纹漂移时正确地拒绝断言防御 delta，都 fail-closed 而非编造结论。再往上，这套 Claim 与 CoverageLedger 已第一次被一个真实消费者读出，也就是面向内部操作者与评估者的报告 view，它把二者投影成 coverage 乘 warrant 与 fidelity 的二维矩阵和一份带边界的报告，刻意不产单一安全总分（做法与安全哲学依据记在 ADR-0017 与 DESIGN 文末的「附」节）。

另外半个卖点，也就是对陌生系统的对抗性故障发现，走出了第一步。一条由操作员提出的日历间接提示注入探针，已经从一段情报编译成可复跑的测量，并在 2026 年 7 月 22 日第一次让它的 security oracle 通过了判别性检验：正对照三十次里命中二十九次，两个同族匹配的负对照各三十次零误报，其中一个负对照把攻击规格原样交给模型而不下达任何执行指令，模型一次也没有照做。这说明这条探针量的是动作服从而不是字符串复制，也就是 ADR-0020 里那条「正对照命中且负对照不误报」的判据首次成立。

这一步的边界必须说清楚。它只说明操作员写法的 oracle 能在一个冻结配置下把正负对照分开，并不说明这条探针可以用于对外保证：重复运行的稳定性尚未验证，`assurance_level` 仍然是 `none`，环境保真度仍然停在 mock，只有模型是真的，结论不外推到任何真实系统。更要紧的是，这套产品假设至今没有被参与开发的人以外的任何人检验过。

这条探针线上还有一次值得记下来的自我更正。此前在另一个模型部署上累计五十次试验都没有观察到一次写入尝试，我们据此提出过一套关于任务形态的机制解释；换部署之后同一条探针稳定点火，于是那批零被重新读作**部署特异**，机制解释相应收回。从单一目标的零去推断机制是一种真实的方法学风险，这个教训记在 `docs/Project_Memory.md`。

由于开发阶段只有一名开发者，探针的每一次计费运行都按 ADR-0022 的「一人两帽」执行：先冻结不可变的执行请求，再在另一个提交里签署批准，然后由 runner 按冻结规则机械执行并留下回执。它提供的是角色分离与可审计性，不提供人员独立性，相关字段在每份产出里都如实标注。

项目仍处切片验证阶段，验证的是这套结论层的形状与纪律站得住，而不是一个打磨完成的成品平台。
