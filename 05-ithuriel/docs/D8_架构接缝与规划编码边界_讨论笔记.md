# Ithuriel D8：架构接缝与规划—编码边界讨论笔记

日期：2026-07-10  
状态：讨论稿，供项目搭档与朋友交流，不是冻结规范

相关材料：

- `docs/architecture-seams-D8.md`

## 1. 讨论背景

Ithuriel 的 D8 是第一条最薄端到端 AI 安全评估切片：针对一个明确的 agent target，使用版本化的间接提示注入场景，比较 bare/defended 两个配置，多次运行后保存原始轨迹、形成统计记录和 Finding，并明确声明范围与未覆盖面。

此前的架构审阅曾建议在 D8 前建设 PlanCompiler、RunOrchestrator、CoverageLedger、Adjudicator 和 Claim Engine 等模块。这一排序随后被否定：这些模块可能是合理的长期目的地，但在单 target、单进程、mock 环境、无并发、无恢复、无跨天审批的 D8 中没有足够摩擦支撑其设计。把它们前置会重新落入“先建平台、后跑切片”的陷阱。

形成的新共识是：

> D8 前固定不可逆的接缝，不建设平台；让最薄切片产生的数据和摩擦决定哪些接缝随后长成模块。

进一步讨论后，又发现“规划期现在固定”仍然不够精确。接口语义、安全不变量、具体字段、序列化算法和长期模块并不应在同一时刻冻结。因此，本文提出三道决策门：

1. 规划期冻结含义、边界和测量有效性；
2. 编码期借助真实 fixture 冻结具体协议、字段和算法；
3. D8 跑通后再决定模块化、平台化和性能优化。

## 2. 对 D8 接缝草案的总体评价

当前 `architecture-seams-D8.md` 已经基本贯彻薄切片原则。合理之处包括：

- Executor 是策略强制点，不能相信上游已经检查；
- 插件按 capability 注册，control 不直接绑定具体工具；
- RawArtifact、Observation、TrialOutcome、Finding 不可压平；
- MeasurementContext 从第一天捕获；
- 证据采用 per-Action manifest，再聚合 run root；
- 探索样本进入确认循环后必须独立重跑或进入 holdout；
- 输出结构化 scope/gap，但不形成完整合规 claim；
- defense 属于被测 target 的配置，而不是评估器内部的防御模块；
- PlanCompiler、Orchestrator、Ledger、Claim Engine 等机器延后。

但在最终冻结前，仍有若干会导致实现产生不同语义的接缝歧义。

## 3. D8 草案中需要修正的关键问题

### 3.1 bare/defended 与“上下文严格相等”的矛盾

bare/defended 的 `defense_hash` 必然不同；如果 `TargetSnapshot` 包含 defense config，那么完整 target snapshot 也必然不同。因此，不能要求两组 MeasurementContext 裸字段全部严格相等。

应引入最小的 ComparisonSpec，明确哪个字段是实验处理变量：

```yaml
comparison:
  kind: defense_delta
  invariant_fields:
    - corpus_version
    - scenario_version
    - target_base_hash
    - model_version
    - attack_strength
    - sampling_plan
    - harness_version
    - detector_version
    - aggregate_rule_version
    - environment_backend
  treatment_field: defense_hash
```

D8 的 compatibility 应表示：除显式 treatment field 外，所有 invariant fields 严格相等。

Target 可拆为：

```text
target_base_hash       = agent/model/tool 的基础配置
defense_hash           = 防御配置；bare 使用 canonical {type: none} 的 hash
target_variant_hash    = H(target_base_hash, defense_hash)
```

这样既保留“defense 属于 target config”的所有权边界，也能表达同一基础系统上的两个实验组。

### 3.2 Backend 不应直接产生 Evidence

Mock、SeededTenant、Sandbox 不应共享 `Action + Evidence` 协议。Evidence 是经过解析和语义解释的领域对象，如果由后端直接产生，不同后端会各自定义证据语义。

更合理的协议是：

```text
EnvironmentBackend.execute(Action)
    → ExecutionReceipt
    + RawArtifactRef[]
    + AuditEventRef[]

RawArtifact
    → Observation / TrialOutcome
    → Evidence
    → Finding
```

还应区分两个维度：

- `environment_backend`：AgentDojo mock、SeededTenant、CustomerSandbox；
- `model_transport`：本地模型、远程模型 API、模拟模型。

AgentDojo 的工具状态是 mock，不代表模型调用也是 mock。远程模型 API 仍有成本、数据出境和速率限制，ModelExecutor 仍然是 PEP。

### 3.3 TrialOutcome 必须区分失败、错误和无效运行

只有 `n_runs`、`n_success` 和 `success_rate` 不足以处理 D8 一定会遇到的 timeout、provider error、detector error 和 retry。

例如，10 次尝试中 8 次 API timeout、2 次攻击失败，不能被计算成 `0/10` 后判 pass。重试也不能被当成新的独立 trial，虚增样本量。

建议最小语义为：

```yaml
trial_outcome:
  - attack_success
  - attack_failure
  - execution_error
  - detector_error
  - invalid

aggregate:
  n_attempted:
  n_valid:
  n_attack_success:
  n_execution_error:
  n_detector_error:
  success_rate: n_attack_success / n_valid
```

需要固定的规则：

- retry 属于同一个 trial 的执行尝试，不增加 `n_valid`；
- 只有 `attack_success/attack_failure` 进入有效统计；
- `n_valid < min_runs` 时不得 pass，应为 `inconclusive`；
- 错误和重试的原始记录必须保留。

### 3.4 校准、测量有效性和防御 utility 必须分开

至少有三类不同结果：

```text
measurement_valid      ← detector/harness 是否正常
security_outcome       ← 攻击是否成功
utility_outcome        ← 防御后正常任务是否仍可完成
```

Instrument calibration 应使用冻结的合成 trajectory：已知攻击成功轨迹必须被检出，正常轨迹不得被误报。这是确定性测试，失败可令 measurement invalid。

端到端 harness 健康检查应使用专门的可控 mock fixture，而不是被评估的 defended target。否则“已知易破样本未命中”可能是防御真的生效或模型随机性，而不一定表示 harness 损坏。

defended target 无法完成正常任务不是 measurement invalid，而是一个有效且重要的 utility regression。否则，一个拒绝所有请求的防御可能被标成“测量无效”，而不是“安全性高但完全不可用”。

还需固定：

> `measurement_valid == false` 的 run 不得产生可进入 rollup 的 pass/fail Finding，只能保留原始结果并输出 inconclusive/invalidated assessment。

### 3.5 现有 schema/profile 与新接缝冲突

目前至少存在以下冲突：

- `ontology_schema.yaml` 仍把 `prev_evidence_hash` 设为必填，与 per-Action manifest/run root 冲突；
- Region Profile 仍使用 `verification.plugin` 直接绑定工具，与 capability 制冲突；
- schema 将 `not_applicable` 描述成 out of scope，与 D1 冲突；
- “tool_version + invocation_params 必须复现 Finding”对非确定性 AI 测试过强，它们只能支持重新执行相同测量协议，不能保证获得相同统计结果。

建议的治理顺序是：

1. pipeline note 决定产品范围和 D1–D8；
2. `architecture-seams-D8.md` 决定 D8 运行时接缝；
3. ontology schema 是已落地的数据契约，D8 数据持久化前必须与接缝同步；
4. 架构图只作解释视图，不作为规范性权威。

D8 只需迁移 `AI-AGENT-PI-01` 到 capability requirement，其他 control 可等对应切片启动时再迁移，不必一次改完。

### 3.6 Action 授权粒度需要在实现前确定

“审批绑定 Action hash”还没有回答 Action 是完整 probe、单个 scenario、一次 trial，还是一次模型调用。

D8 可考虑：

```text
ProbeAction
  ├─ target_variant_ref
  ├─ capability
  ├─ scenario_set_ref
  ├─ n_trials
  ├─ sampling_plan
  ├─ token/cost/time budget
  ├─ environment_backend
  ├─ detector_ref
  └─ aggregate_rule_ref
```

每个 trial 是 ProbeAction 下的 TrialRecord，不是独立审批对象。未来真实环境中，如果 agent 动态提出具有真实副作用的子动作，该子动作仍必须再次经过 PEP。

还需固定：

- preflight 接受已经完整物化但尚未授权的 Action；
- pre-dispatch 检查同一不可变 Action、审批引用、当前策略版本和 target snapshot；
- policy-relevant 字段变化会改变 hash，并使原审批失效；
- secret 使用 secret reference/version，不进入 canonical Action 明文；
- canonical serialization 和 digest algorithm 带版本；
- 使用测试向量保证不同实现对同一个 Action 得出同一 hash。

## 4. 其他值得保留的建议

### 4.1 Canary detector 不能只检查字符串出现

模型可能在拒绝攻击时引用或复述 canary。一个确定性的 substring detector 仍可能稳定地产生错误结果。

优先使用：

- 结构化 forbidden tool call；
- AgentDojo state delta；
- 指定字段或 channel 中的 per-trial nonce；
- nonce 与预期动作同时满足的组合条件。

### 4.2 MeasurementContext 应为结构化对象

`scenario_id/version`、`model_id/version` 适合说明文档，不适合作为机器字段。最终应使用嵌套对象，并包含 context schema version 和 sampling plan，例如：

```yaml
context_schema_version:
sampling_plan:
  n_trials:
  seed_schedule:
  order_policy:
```

### 4.3 scope/gap 先定义最小种子，不建 Ledger

```yaml
control_id:
target_ref:
applicability:
execution_status: tested | not_tested | unsupported | out_of_scope
reason:
action_refs: []
assurance_level: none
```

`assurance_level` 更适合放在 run/report 或 scope statement 上，而不是 Finding 字段。

### 4.4 Defense config 不能自动消除独立性问题

将 defense 放入 target config 是正确的架构边界，但评估者仍可能选择、调参或部署某个防御。建议记录：

```text
provider/owner
source_ref
version/commit
config_hash
provisioned_by
relationship_to_evaluator
```

D8 使用第三方已发表防御并披露来源即可，不需要为此建设治理模块。

### 4.5 架构图应是解释视图，不是唯一规范

Claude artifact 不适合充当长期、可离线审阅的权威来源。运行时 v2 图应导出并提交到仓库，同时明确：图解释规范，但不替代 schema、决策记录和接缝契约。

## 5. 哪些必须在规划期严格确定

规划期决定“什么必须始终为真”。满足以下任一条件，就应在规划期冻结：

- 会改变产品可以声称什么；
- 会改变什么算 pass/fail 或有效测量；
- 会改变信任、授权或副作用边界；
- 两种实现会让同一结果具有不同含义；
- 会使历史 Evidence 无法解释，而不仅是无法解析；
- 会改变哪些东西由 Ithuriel 拥有、哪些属于被测目标。

### 5.1 D8 范围和非目标

- 只评估一个明确的安全 control；
- mock 指环境工具状态，不自动表示模型调用也是 mock；
- 输出 Finding 和结构化 scope/gap；
- 不形成完整合规 claim；
- 不建设编排、并发、恢复和服务化平台。

### 5.2 信任和副作用边界

- 插件不能直接产生副作用；
- Executor 是 PEP；
- 上游检查不能代替执行前检查；
- 授权只能覆盖明确、有界的动作；
- Action 变化后原审批失效；
- 新产生的真实副作用必须再次经过 PEP；
- ModelExecutor 同样执行成本、端点和数据策略。

### 5.3 所有权与独立性边界

- defense 属于 target variant/config；
- Ithuriel 不把被评估防御实现成评估核心的一部分；
- 工具实现可以借用，领域契约必须自己掌握；
- defense 来源及其与评估方的关系必须披露。

### 5.4 数据语义边界

以下概念不能压平：

```text
RawArtifact → Observation → TrialOutcome → Aggregate Finding
```

同时必须区分：

- retry 与新 trial；
- execution error 与 attack failure；
- measurement invalid 与 security pass；
- utility failure 与 measurement invalid。

### 5.5 测量和比较有效性

- bare/defended 只有一个显式 treatment variable；
- 其他受控条件必须满足 compatibility rule；
- 归一化不自动意味着可比较；
- 探索样本不能同时证明自身稳定；
- detector/harness 校准、security outcome、utility outcome 分开；
- 没有足够有效 trial 时不能 pass。

### 5.6 状态词汇的含义

规划期必须明确以下词汇分别属于 run、trial、coverage 还是 Finding，并决定其覆盖度语义：

- `not_applicable`
- `not_tested`
- `unsupported`
- `out_of_scope`
- `inconclusive`
- `measurement_invalid`
- `execution_error`
- `detector_error`

### 5.7 证据完整性的总体形状

- 每个 Action 独立形成 content-addressed manifest；
- run 聚合各 Action manifest；
- 不采用全局线性 `prev_evidence_hash` 作为长期模型；
- manifest 引用原始材料、执行事实、策略、规则和上下文；
- hash 和签名只能证明完整性，不能证明内容真实性。

## 6. 哪些应在开发编码阶段确定

编码阶段用第一批真实 fixture 塑造具体协议。每项决定应以 schema、ADR、golden fixture 或测试向量落地。

判断标准是：

- 需要看到 AgentDojo trajectory、模型错误或实际 raw output 才能合理设计；
- 可以在不改变规划语义的前提下替换；
- 主要影响解析、兼容性、维护成本或性能；
- 必须在第一次保存持久数据或公开接口前确定。

| 主题 | 规划期冻结 | 编码阶段确定 |
|---|---|---|
| Action | 授权必须绑定有界副作用 | ProbeAction 字段、trial 子记录结构 |
| Action hash | 必须规范化、版本化、修改即失效 | canonical serialization、hash 算法、字段包含规则 |
| PEP | preflight + pre-dispatch | 函数签名、decision 对象、错误类型 |
| Backend | 只返回执行事实和 raw | `execute()` 接口和 ExecutionReceipt 字段 |
| Capability | control 不直接绑定工具 | capability ID、descriptor schema、匹配代码 |
| MeasurementContext | 捕获影响可比性的条件 | 嵌套字段、context hash、sampling plan |
| Comparison | 只有显式 treatment 可不同 | ComparisonSpec 的机器格式 |
| TrialOutcome | 成功、失败、错误、无效分开 | enum 名称、计数字段、retry 记录 |
| Detector | D8 使用确定且可重放的 detector | canary/state-delta 算法和 fixture |
| Aggregation | 只对有效 trial 聚合 | 错误计数、容差和版本字段 |
| Calibration | measurement validity 与 utility 分开 | fixture、阈值、最小运行次数 |
| Evidence | per-Action manifest → run root | manifest schema、排序、去重和 root 算法 |
| Scope/gap | D1 语义且不形成 claim | record 字段和输出格式 |
| Target snapshot | base 与 defense variant 分开 | 序列化、secret reference 和 hash 结构 |

一个特别重要的编码决策门是：

> 在写 Executor 主路径前，先用一条真实 AgentDojo trajectory 作为 fixture，再以 ADR 确定 `ProbeAction → TrialRecord → ExecutionStep` 的层次。

这个粒度不应在规划期凭空设计，也不能等 Executor 写完后再补。

## 7. 哪些应等 D8 跑通后再决定

以下事项应根据真实运行时间、错误、数据量和人工操作摩擦决定：

- 是否需要独立 PlanCompiler；
- 是否需要 RunOrchestrator；
- 是否需要完整 CoverageLedger；
- 是否需要 ExperimentManager；
- Claim/Assurance Engine 的实现；
- Temporal、消息队列和分布式 worker；
- 并发 DAG、流式存储和缓存；
- PostgreSQL/对象存储的最终部署形态；
- OPA 是嵌入、sidecar 还是先用本地确定性函数；
- Rust 或 Go 下沉；
- 通用 compatibility predicate；
- LLM judge；
- 统计置信区间、效应量和功效分析；
- 签名、可信时间戳和透明日志。

这些问题主要决定机器“长成什么样”，不应阻塞第一条最薄切片。

## 8. 建议的文档结构

`architecture-seams-D8.md` 可以从当前的桶 A/桶 B 进一步拆成三部分：

1. **规划期冻结的不变量**：使用 MUST/MUST NOT 描述语义、安全和测量边界；
2. **D8 编码决策门**：在首次持久化/API 冻结前，通过 fixture、ADR、schema 和测试向量决定；
3. **D8 后延后机器**：保留 PlanCompiler、Orchestrator、Ledger、Claim Engine 和性能平台等长期方向。

这套结构的口诀是：

> 规划期冻结含义、边界与有效性；编码期冻结协议、字段与算法；切片后再决定模块、平台与性能优化。

## 9. 可用于讨论的问题

1. `ProbeAction` 应是一次完整 probe、一个 scenario，还是一组有界 trials？
2. defense 属于 target config 后，哪些评估者行为仍然构成独立性风险？
3. 对模型 API timeout 和 provider refusal，什么条件下应判 inconclusive，什么条件下应判 measurement invalid？
4. utility regression 应是独立 Finding、实验指标，还是 defense-delta 报告中的共同主结果？
5. MeasurementContext 的哪些字段必须严格相等，哪些可以由 ComparisonSpec 声明为 treatment？
6. 一个 content-addressed manifest 能证明哪些事情，不能证明哪些事情？
7. capability 应表达技术能力、测量能力还是标准 control 的验证能力？
8. D8 第一次真实 fixture 出现后，哪些当前假设最可能被推翻？

## 10. 小结

这次讨论的核心不是“设计多一点”或“设计少一点”，而是把不同类型的决定放在正确的时间点：

- 安全、授权、测量有效性和产品 claim 的语义不能等编码时临时决定；
- 字段、序列化、接口签名和算法需要真实 fixture 塑形；
- 模块、服务、持久化工作流和性能优化必须等待实际摩擦。

这样既能防止薄切片演变成平台建设，也能避免第一批数据在安全语义和测量语义上留下不可修复的错误。
