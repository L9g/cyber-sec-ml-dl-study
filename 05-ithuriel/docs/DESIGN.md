# Ithuriel 项目原理与编码介绍

> 名字取自弥尔顿《失乐园》里那杆一触即现原形的矛。项目做的正是同一件事：用探测让 AI 系统与传统系统里隐藏的失败现形，再把探测结果凝成经得起复核的结论。
>
> 本文面向外部读者与开源社区，默认你没有读过本仓库的架构决策记录（ADR）或内部讨论。读完你应当能回答三个问题：这个项目为什么存在，它的核心纪律是什么，代码是怎么围绕这套纪律组织起来的。

---

## 0. 一句话：是什么，不是什么

**是什么。** 一个以保证（assurance）与对抗性故障发现为主线的安全合规 Agent。它把安全标准蒸馏成机器可读的 ontology，驱动现成的扫描与探测工具去测一个目标系统，再把工具的原始输出凝成一份可审计、可复现、带明确边界的结论。

**不是什么。** 它不是又一个安全扫描平台。扫描与探测能力全部借用成熟工具，项目自己只投入两样东西：把标准蒸馏成 ontology，以及承载证据、Finding 与保证结论的那一层。价值在结论层，底座只是被审计的对象和执行手段。

这条边界是理解后文一切设计的前提，所以放在最前面讲清楚。

---

## 1. 动机：为什么需要一个"保证层"

agentic 安全与 AI 保证这个领域，眼下最缺的不是工具，而是从"跑了一个工具"到"能对一个陌生系统下一个有边界、可复现、可审计的结论"之间的那一层承载物。

几个来自领域文献的信号坐实了这个缺口。安全评测的 benchmark 高度碎片化，大量结果只跑一次，缺少可复现的测量上下文，换个种子或换个镜像分数就漂移。多篇工作反复指出，模型对齐本身并不保证 agent 安全，因此必须以 assume-breach 的姿态去设计评测。换句话说，光有一堆扫描器和 red-team 工具还不够，真正稀缺的是把它们的输出变成"结论"的规程：这个结论在什么测量条件下成立，它的证据能不能被别人重放，它到底断言了什么、又刻意没有断言什么。

Ithuriel 想补的正是这一层。工具照借，结论层自建。

## 2. 核心构建纪律：两层模型（借底座，建差异化）

整个项目由一条纪律统摄，理解了它，后面每个模块的取舍都顺理成章。

**Base 层等于借。** 执行机制、进程间通信、调度、以及所有扫描与探测工具，都借用成熟实现，能借就不自建。AI 注入探测借 AgentDojo 与已发表的防御，端口探测借 nmap，防火墙配置读取借系统自带的 `ufw`，注入检测器借 `transformers` 生态里现成的 prompt-injection 分类器。这些东西对本项目而言只是"可信的被审计对象"或"执行手段"，重写它们没有意义。这条边界要划精确一点：借的是执行**机制**，但把执行**接缝**建成一个强制执行点却是自己的事。带副作用的动作要经过 `executor.py` 那个两阶段的策略执行点（PEP），由它强制 RoE 授权、命令白名单、以及派发前的独立复检。那套安全与保证语义是差异化层建的，不在借用之列。

**Differentiator 层等于建。** 项目只在两处刻意投入。第一处是把标准蒸馏成 ontology，也就是把 Cyber Essentials、NIST CSF、OWASP LLM Top 10 这类规范，落成机器可读、可被代码消费的控制与标准注册表。第二处是证据、Finding 与保证层，也就是本文第 3 节讲的那组对象。

**插件是薄适配器。** 每个插件的职责只是调用现成工具，再把它的输出归一化进统一的证据 schema，绝不为某个控制手搓专用扫描逻辑。缺工具的控制不硬凑，如实留成覆盖缺口。

有一条护栏要强调：minimal base 并不等于 toy base。底座要借到"领域内行点头"的程度，不是随便糊一个玩具就算，只是不多建一分自己不该建的东西。

## 3. 概念模型：差异化层的核心对象

这一节介绍"建"的那半边到底建了什么。所有对象都定义在 `src/ithuriel/models.py`，用 pydantic v2 写成，字段形状是据真跑数据的真实摩擦反推出来的，而不是照着论文一次设计齐。

**三级证据分离：Raw、Observation、Finding。** 工具吐出的原始文本是 Raw，把 Raw 解析成结构化观察是 Observation，对一个控制在一个目标上的裁定结果才是 Finding。三者不许压平成一个东西，因为它们的可信度与复现要求各不相同。

**Finding 四态。** 一个 Finding 的 `status` 只能取四个值，每个值都有严格语义：

- `pass`，控制满足。
- `fail`，控制不满足，此时必须同时带上 `rationale`（理由）与 `severity`（严重度），否则构造直接报错。
- `not_applicable`，控制对本目标不适用，移出覆盖率分母，同样必须带理由说明为何不适用。
- `inconclusive`，证据不足以裁定，需要重采或升级人工，必须带理由。

这些约束不是文档里的君子协定，而是写进 pydantic 校验器的硬规则。构造一个没有理由的 `fail`，程序会当场抛错。

**verdict_mode 三值。** 裁定是怎么做出来的，与裁定结果本身正交。`automatic` 指确定性检测器，`llm_judge` 指用大模型当裁判，`human_review` 指人工复核。这三者是三条不同的裁定来源。

**AiRunRecord。** AI 探针是非确定的，同一个攻击跑十次结果可能不同，因此每条 AI Finding 挂一份运行统计：模型标识、版本快照、温度、种子、总尝试次数、有效次数、执行错误次数、成功次数、成功率，以及 Wilson 置信区间。确定性检查没有这份记录，此时 `run_record` 就是 `None`，后面会看到这个 `None` 承载了很重要的语义。

**ComparisonSpec：delta 不是一条 Finding。** 防御效果的度量，比如"加了检测器之后注入成功率下降了多少"，本质是跨两条 Finding（裸跑与加防御）的一次比较，因此单独建模成 `ComparisonSpec`，而不是塞进某条 Finding。这个比较对象只承载 delta，不重算裁定，并且强制成对报告 security 与 utility 两轴，不许只报安全轴。

**joint_verdict：security 与 utility 的联合裁定。** 一个防御可能把注入压下去了，代价却是把目标的正常任务能力也一起废掉。只看安全轴的 Finding 会把这种情况误报成"通过"。因此比较层带一个恒有值的联合裁定字段，取值为 `acceptable`、`security_failed`、`utility_failed`、`inconclusive` 之一。它独立地从原始输入计算，刻意不去读那个会随实验调整阈值的 tradeoff 分类，避免用一个不稳定的分类法反过来控制裁定。

**EvidenceManifest：内容寻址的证据清单。** 每条 trial 的原始记录按内容哈希编址，汇聚成一个 run root 哈希。原始证据不可被覆盖、不可被压平。只要输入相同，run root 就稳定复现，这是 bit 级可复现的契约。

**ScopeStatement：把边界写进结构。** 每份报告都带一个范围声明，里面钉着一句 `assurance_level: none`。这是为了防止一次 demo 被误读成"合规通过"。范围声明同时列出哪些东西进了覆盖率分母、哪些没覆盖，它也是后面 CoverageLedger 的种子。

**Registry：标准注册表的牙齿。** 控制引用标准（比如某控制引用 OWASP LLM01），而每个被引用的标准来源必须在 profile 的 standards 注册表里真实声明过。加载注册表时有一个校验器专门检查这件事，一旦发现悬空引用就抛错。这条"schema 不变量"因此长了牙，而不只是一句规范。

最后一层信封是 `AssuranceReport`，它把一格真跑的全部"建"产物打包：测量上下文、证据清单、若干 Finding、若干比较、范围声明，以及解析后的控制与其引用到的标准。跨多次测量条件的一整个会话，再往上打包成 `SessionReport`。

**verdict_provenance、Claim 与 confidence_basis：标准到保证结论的上半边。** 前面这些对象承载的是"证据与 Finding"这半边，还有一半是"从证据到结论"。裁定溯源被升成一个带类型的字段 `verdict_provenance`，它把 `verdict_mode` 里那个含混的 `automatic` 拆开：AI 统计探针、确定性配置规则、带执行的主动探测在旧模型里都叫 `automatic`，但它们的置信度体制并不一样，因此用一个可辨识联合来区分：`AutomaticRuleProvenance` 带一个 `measurement_kind`（统计试验还是确定性观察），`HumanAttestationProvenance` 则指向人工裁定的证据与映射版本。在它之上是一个纯函数的 Claim 层（`claim.py` 的 `derive_claims`），把每条 Finding 消费成一条 `Claim`，回答的是"这个结论凭什么成立、又只对什么范围成立"。凭据不是一个压扁的单一置信度分数，而是一份多维的 `confidence_basis`（warrant）：裁定机制、不确定性体制、可复现级别、目标保真度，外加一份诚实的局限清单。这一步刻意 fail-closed：报告里没有 Finding 就返回空列表而不是一个正向 Claim，Finding 缺 `verdict_provenance` 就标成不可评估而不给乐观档。它是差异化层"标准到保证结论"的最薄上半边，只做纯函数，不建引擎。

## 4. 核心不变量：本项目的牙齿

如果说第 3 节是"有哪些对象"，这一节就是"这些对象凭什么可信"。下面每条不变量都配一个"不这么做会怎样"的反例，因为反例最能说明它们为什么存在。

**诚实闸门 assertable。** 一个 defense delta 能不能被断言，由一个合取条件决定：

```
assertable = measurement_valid  and  (not underpowered)  and  (not confounded)
```

`measurement_valid` 要求测量本身有效（比如存在正对照、没被配额截断），`underpowered` 要求置信区间不重叠到看不出差异，`confounded` 要求两臂之间除了防御这个处理变量之外没有别的未声明差异。任何一条不满足，harness 就拒绝报出 delta，只如实标出失效原因。反例：如果没有这道闸门，一次置信区间大幅重叠的跑也会被报成"防御有效 delta 负零点一"，读者会把纯噪声当成结论。

**裸跑成功率为零应当是 inconclusive，而不是 pass。** 这是全项目最反直觉、也最能说明立场的一条。如果一个注入攻击在没有任何防御的裸目标上成功率就是零，正确的结论是"无定论"，而不是"目标安全"。因为成功率为零很可能只是说明这个攻击变体太弱、或者这个目标恰好对这一族攻击鲁棒，缺少正对照的情况下推不出安全性。反例：把裸跑零成功率当 pass，会让一个根本没被有效攻击过的目标拿到一张"安全"的假证书。

**unsupported 区别于 not_applicable，两者去向相反。** 控制没有对应工具去测，是 unsupported，它进覆盖率分母，记为一个覆盖缺口。控制对这个目标根本不适用，才是 not_applicable，它移出分母。反例：把"没工具测"误标成 not_applicable，会把一个真实的覆盖缺口从分母里抹掉，让覆盖率虚高。

**security 与 utility 必须成对裁定。** 见第 3 节的 `joint_verdict`。反例：只报安全轴，一个"检测到注入就直接中止任务"的防御会显得完美，实际上它是拿可用性换安全，任务一个都没做完。这个"中止任务到底算不算安全"的追问牵出的是一整套安全哲学立场，本文末尾的"附"一节专门摊开它，也解释了它如何决定报告的呈现方式。

**fail-closed 比较。** 一旦置信区间重叠，或者两臂的溯源出现漂移（比如同一个滚动别名在两次跑之间指向了不同的模型快照），harness 一律拒绝断言，标注 underpowered 或 context invariant mismatch，绝不编造一个 delta 数字。反例：溯源漂移不拦，等于把两个不同模型的分数相减，却声称测的是防御效果。

**内容寻址的稳定哈希。** Finding 的 id 由裁定性字段的内容派生，证据清单的 run root 由全部原始 artifact 派生。相同输入得到相同哈希，因此结论可以被第三方重放核对。反例：用时间戳或自增序号当 id，两次相同的跑会得到不同的标识，复现核对无从谈起。

## 5. 跨域通用性：四种裁定形状，四个控制，零 schema 改动

差异化层最强的一个论证，是同一套对象模型能承载形状迥异的四类裁定，全程没有为任何一类新增或修改 schema 字段。这四条端到端切片如下。

| 切片 | 控制 | 裁定形状 | 域 | severity | verdict_mode |
|------|------|----------|-----|----------|--------------|
| slice 1 | AI-AGENT-PI-01 | 非确定 AI 探针（mock 环境） | ai_agent_security | High | automatic |
| slice 2 | CE-UK-FW-03 | 确定性配置读取 | network_security | Medium | automatic |
| slice 3 | CE-UK-FW-01 | 主动探测，带执行与授权机器 | network_security | Medium | automatic |
| slice 4 | CE-UK-SU-03 | 人工复核，声明式证据 | 安全更新治理 | Low | human_review |

这四条切片横跨三个域、三个严重度档、两种裁定模式，涵盖了非确定的统计探针、确定性的配置读取、带副作用的主动探测，以及完全没有工具产出的人工声明。它们全部由同一套 Finding、Evidence、Assurance、CoverageLedger 承载。

关键观察在于那些"没有值"的地方恰好被优雅容纳。确定性检查没有 AI 运行统计，`run_record` 就是 `None`。配置读取不是一次防御对照实验，`comparisons` 就是空列表。人工复核的证据是一条结构化声明记录，而不是工具文本。这些情况没有一个逼出新字段，说明这套模型的抽象层次选对了。

要强调的正是这四个字：**零 schema 改动**。四种裁定形状在直觉上差得很远，一个是非确定的统计探针，一个是人工签字的声明，中间还夹着确定性配置读取和带副作用的主动探测，可它们落进同一套对象时，没有一处需要给 schema 动刀。这才是这一节真正的分量所在。

需要同时说清楚的是，这句话不等于"schema 从此冻死永不改"。恰恰相反，本项目的纪律是据真实摩擦演进 schema，摩擦到了就改，第 7 节会专门讲这条方法学。这里的成就因此要精确地读：不是 schema 不可变，而是这四种**裁定形状本身**没有逼出任何改动。真正逼出一次改动的，是后来在它们之上建一个消费者（Claim 层）的时候，第 7 节会讲那次改动是被什么逼出、又是怎么按纪律加进去的。

还有一处保真度上的精确要说清楚：表里 slice 1 标的"mock 环境"，指的是 AgentDojo 那个工具环境是 mock，并不代表模型也是假的。环境保真度和模型传输是两根正交的轴，一个 mock 的工具环境完全可以配一个真实的远程模型 API，照样产生真实的 token 成本与数据出境。后面第 8 节讲运行时会再回到这一点。

## 6. 代码导览与端到端走查

整个"建"的部分大约两千行 Python，分布在十二个模块里。按"越核心越靠前"排列：

- `models.py`，schema 主体，所有对象与不变量校验都在这里。
- `derive.py` 与 `derive_session.py`，把 AI 真跑的扁平 JSON 反推成结构化 Finding 与会话报告。
- `registry.py` 与 `capability.py`，读 profile 里的控制与标准注册表，并做一次能力匹配。
- 四个薄适配器 `config_inspection.py`、`port_scan.py`、`executor.py`、`attestation.py`，各对应一类裁定形状。
- `ledger.py` 与 `provenance.py`，跨控制的覆盖率 rollup，以及溯源钉死。
- `claim.py`，Claim 层，把 Finding 消费成带 `confidence_basis` 的保证结论（纯函数，不建引擎）。

下面用 slice 2（防火墙默认拒绝策略的配置读取）走一条完整链路。之所以选它，是因为它确定性、不需要任何 API key、可离线复现，而且源码里带了大量 `# FRICTION:` 注记，正好顺带展示第 7 节要讲的方法学。

这里要说清楚一件事，免得误会：选 slice 2 只是因为它教学起来最干净，它并不是这个项目的招牌。项目真正的招牌形状是第 1 节动机里那条 AI 红队探测，也就是对一个陌生 AI Agent 冷启动地把注入漏洞打出来。防火墙配置读取扮演的角色是一道证明题，用来说明同一套保证层连确定性检查也能干净地承载。招牌负责吸引注意，而这套保证层跨越四种形状的通用性才是它真正站得住脚的地方，两者不要混为一谈。

链路一共五步，全在 `config_inspection.py` 的 `build_report` 里串起来。

**第一步，能力匹配。** 控制 `CE-UK-FW-03` 声明它需要一项能力 `host.firewall.default_policy.inspect`，`ufw` 适配器声明它提供这项能力。匹配就是一次集合包含判定：

```python
def adapter_satisfies(control_id, adapter):
    reqs = required_capabilities(control_id)
    return bool(reqs) and reqs <= adapter.provides
```

这里刻意不是一个 planner，没有候选排序、没有参数规划、没有插件注册中心。一个控制、一项需求、一个适配器、一次子集判定，仅此而已。匹配不上就返回一份覆盖缺口报告，不产 Finding。

**第二步，解析 Raw 成 Observation。** `parse_ufw_status` 是纯函数，把 `ufw status verbose` 的原始文本抽成三个字段：防火墙是否激活、入站默认策略是什么、默认策略行是否存在。它只认这两行核心证据，不解析其余规则明细，因为这一格只裁默认策略。

**第三步，确定性版本化规则做裁定。** `evaluate_default_deny` 把 Observation 映射成 status 与理由。这里体现了第 4 节那条不变量的同源性：如果 UFW 处于 inactive，函数返回的是 `inconclusive` 而不是 `fail`，因为 UFW 没激活推不出主机没有默认拒绝策略，很可能是 nftables 或云安全组在执行。只有当目标快照明确声明 UFW 是唯一权威执行面时，才会判 `fail`。缺少正信号不等于反结论，这与 AI 切片"裸跑成功率为零得到 inconclusive"是同一条认识论纪律。

**第四步，组装成 Finding，观察 None 是否被容纳。** 关键就在这一行：

```python
finding = Finding(
    ...,
    run_record=None,   # 确定性检查无 AI run
    root_causes=None,  # root_cause 是 AI 机理枚举，对防火墙不适用
)
```

一个为 AI 探针设计的模型，被一个确定性检查复用，`run_record=None` 自然地容纳了这种情况，没有逼出任何新字段。这正是第 5 节跨域论证的微观证据。

**第五步，打包 AssuranceReport 并写范围声明。** 因为这不是一次防御对照实验，`comparisons` 是空列表，语义完整。报告同时挂上解析后的控制与它引用到的标准（Cyber Essentials 与 NIST CSF），审计闭环因此闭合：从 Finding 的 `control_id`，到控制的 `standards_refs`，到每个 source 在注册表里的真实条目。

**测试哲学。** 项目有两层测试，刻意不混写。确定性的代码契约用 pytest 单测，贴死精确值与边界。与模型分数相关的叙事回归只用带 margin 的方向性断言，禁止硬阈值，因为分数会随种子、版本、镜像漂移。目前一共 150 个测试函数，确定性切片的部分完全离线可跑，不需要任何外部依赖。

## 7. 方法学：据真实摩擦定 schema

这个项目在 schema 设计上有一条贯穿始终的纪律：先跑最薄的端到端切片，据真跑暴露的真实摩擦再决定加什么字段，绝不提前照着论文把每个字段一次设计齐。

原因是提前设计的字段十有八九是错的。真实数据的结构、真实的评分困难、真实的复现问题，只有在跑通一格之后才暴露出来。因此 `models.py` 开头就写明，它只取 ontology schema 的必需字段加上真跑数据真正用到的字段，那些冻结的或搁置的字段（比如 threat_model、fidelity_gap、llm_judge）一概不加。

有一个案例最能说明这条纪律的力度，它关于一个字段：裁定溯源。它先是被刻意按住了很久，然后在恰当的时机被一个真实消费者逼了出来。两个阶段合起来，比任何一边单独都更能说明这条方法学。

背景是这样。第 3 节说过，裁定有三条来源：确定性规则、AI 运行、人工复核。做到第四条切片时可以看到，四种裁定形状各自把裁定来源塞在不同的地方：只有 AI 切片有一个带类型的家可放（就是 `ai_run_record`），另外三种（规则版本号、执行回执、复核记录）全都塞进了 `measurement_context` 这个自由字典里。一个统一的裁定溯源字段呼之欲出，这个信号已经被四条切片反复确认了四次。

按很多项目的习惯，看到这么强的信号就该立刻抽象。但当时刻意没有加。理由是：那时没有任何一个下游消费者因为缺这个字段而断裂，CoverageLedger 只读 status、verdict_mode、severity，四种形状全都跑得通。据真实摩擦定 schema 的纪律，要求的正是这种克制：信号再强，只要还没有一个真实的消费者被它逼到必须区分裁定来源，就不加。

后来那个消费者真的出现了，字段也就随之加了进去，但加的方式本身又是一次纪律的演示。逼出它的是第 3 节末尾讲的 Claim 层：一个要按裁定"怎么得来"赋予不同凭据的消费者，没有这个结构就只能去翻那个自由字典里各形状不同的键，那段提取脆弱得一碰就碎（举个真实的坑，主动探测和配置读取共用了同一个键，靠翻字典去认，顺序稍错就把探测误当成配置读取）。这就是把它逼出来的真实摩擦。加的时候有两个刻意的选择。其一，不加一个扁平的四值枚举，而是加一个带类型的可辨识联合 `verdict_provenance`，因为那四个候选值并不正交：AI、配置、探测三者的裁定**机制**其实都是"确定性规则作用在测量结果上"，真正不同的是下面的测量**体制**，只有人工复核才是另一种裁定权威。其二，消费端产出的凭据不是一个压扁的单一置信度档，而是一份多维的 warrant，因为把覆盖率、证据质量、目标保真度压成一个数字，会重新制造出"绿灯但其实没真测"的问题。整个改动落在 pydantic 的 advisory 层，被只读消费的那份 ontology schema 至今一个字段都没动过。

先把"看到强信号仍然按兵不动"、再把"直到真实消费者逼出才动手、且动手时仍守住形状与克制"都写进文档，是想让读者看清这条方法学不是一句口号。项目的全部十六份架构决策记录（ADR 0001 到 0016）连成一条完整的决策弧，记录了每一次是据什么摩擦、加了或没加什么字段，它们是这条纪律可被追溯的证据。

## 8. 状态与如何运行

**当前状态。** 四条切片全部合入主干，测试全绿，一共 150 个测试函数。差异化层已经跨四种裁定形状、四个控制验证过跨域通用性，跨控制的 CoverageLedger 也已经起步。在这之上，"标准到保证结论"的上半边落了最薄的一层：typed `verdict_provenance` 加纯函数 Claim 层，产出多维 warrant（见第 3、7 节）。这一层还用真实模型跑通验证过一次：多目标真跑里，系统在没有正对照时拒绝把裸跑零成功率当"安全"，在两臂溯源指纹漂移时拒绝断言防御 delta，都 fail-closed 而不编造结论。需要说清楚的是，项目仍处在切片验证阶段，它验证的是这套结论层的形状与纪律站得住，而不是一个打磨完成的成品平台。已落的是最薄的纯函数 Claim 层，完整的 Claim 与 Assurance 引擎（要处理适用性、claim 主体、时间有效性、评估者独立性这类治理，而不只是多个 Finding 的 rollup）连同横向铺开更多控制、场景与攻击族，都还在后面的路线图上。

**环境。** 用 `uv` 管理依赖，Python 需要 3.11 以上。同步环境后即可跑测试：

```bash
uv sync
.venv/bin/pytest
```

确定性的切片（配置读取、端口探测、人工复核）完全离线，不需要任何外部服务，直接消费仓库里冻结的 fixture。

**AI 切片。** 非确定的注入探测切片需要一个能做工具调用（function calling）的大模型。harness 与 provider 无关：通过环境变量 `D8_PROVIDER` 选一个供应商，再把该供应商的 key 放进它约定的环境变量里即可。目前内置了 OpenAI、Gemini、Groq、OpenRouter、DeepSeek、Together、Mistral、Anthropic、Cohere 等预设，也支持本地模型与任意兼容端点。

一般模式是"选供应商，给对应 key，指定模型"：

```bash
# 通式：换 D8_PROVIDER 与它对应的 <PROVIDER>_API_KEY 即可
D8_PROVIDER=<provider> <PROVIDER>_API_KEY=<key> D8_MODEL=<model> \
  .venv/bin/python scripts/run_bare_vs_defended.py

# 示例：某聚合入口
D8_PROVIDER=openrouter OPENROUTER_API_KEY=<key> D8_MODEL=<model-slug> ...

# 示例：本地模型，完全无需 key
D8_PROVIDER=ollama D8_MODEL=llama3.1 ...   # 先 ollama serve

# 示例：任意 OpenAI 兼容端点
D8_PROVIDER=custom D8_BASE_URL=<url> D8_API_KEY=<key> D8_MODEL=<model> ...
```

如果环境里已存在某个已知供应商的 key，脚本还会自动探测选用，`D8_PROVIDER` 也可省略。选模型时唯一的硬要求是它能稳定做工具调用，因为 AgentDojo 的 agent 环回依赖 function calling。

跑起来后，它会做一格裸目标对加防御目标的对照，产出扁平 JSON，再由 `derive.py` 反推成结构化的 Finding 与比较。要留意的是 harness 全程如实报告测量是否有效，从不为了凑出一个好看的 delta 而编造数字，这正是第 4 节那些不变量在真跑里的落地。

## 附：检测到注入就中止任务，算不算安全？

这一节把一个反复出现的追问单独摊开，因为它触到本项目的立身之本，而不是某个实现细节。场景很具体：拿一个 AI agent 去试提示词注入，加了一个注入检测器之后，检测器一嗅到注入就把整个任务直接中止了。结果是攻击一来，agent 没有被劫持去干坏事，但它也不再做本该做的正事。那么，这个系统算安全吗？

诚实的回答是，这一问本身就问错了。安不安全不是一个能用一个字回答的属性，中止任务在一根轴上是安全，在另一根轴上正是不安全，而想把它拍成一个整体的“是”或“否”，恰恰是这个项目要拦住的动作。下面一步步拆。

**“安全”其实是好几个不同的属性捆在一起。** 至少三个要分开看。第一是完整性，也就是攻击者有没有让 agent 去干坏事，比如外泄数据、发未授权邮件、执行注入进来的命令。在这根轴上，中止等于成功，因为它拒绝了注入指令，什么坏事都没做。第二是可用性，也就是攻击来了以后 agent 还做不做它的正事。在这根轴上，中止等于失败，因为攻击者用一段注入字符串就让它罢工了。第三是无害，也就是系统有没有造成伤害，中止没有做出任何动作，这一条上是干净的。所以“它安全吗”要看你问的是哪一条，在完整性与无害上它扛住了，在可用性上它塌了，用一个词答不了。

**别忽略：攻击者其实赢了一样东西。** 就算没被劫持，攻击者也把一段本该只是数据的不可信内容，比如一条日历事件描述或一封邮件的正文，变成了对 agent 可用性的控制。这是一种真实的、与安全相关的能力，即注入触发的拒绝服务。所以这并不是“没问题、完全安全”，系统有一个实打实的弱点，就是不可信内容能让它停摆。这个弱点要不要紧是另一回事，但它是弱点，跑不掉。

**这其实是可靠性工程里的老问题。** 用 RAMS 的语言说，这就是 fail-safe 与 fail-operational 之分。Fail-safe 是出故障就进入安全态，通常是“停”，比如信号失效时列车刹停，安全但不可用。Fail-operational 是出故障还得继续工作，只是降级运行，比如飞机飞控就必须这样，因为“停”本身要命。中止注入是 fail-safe，不是 fail-operational。而 fail-safe 到底够不够安全，取决于“失去功能”这件事本身算不算一种危害，这跟“绿灯坏了该闪红还是全灭”是同一类判断，没有脱离场景的正确答案。

**于是“算不算安全”是一个风险与策略判断，不是一个技术事实。** 对一个高危系统，比如 agent 能动钱、能发邮件、能执行代码，“宁可罢工也绝不被劫持”往往正是你想要的安全属性，这时中止就是对的、安全的行为。而对一个可用性即使命的系统，比如医疗分诊或应急响应，一段注入就能把它拒绝服务，这个中止就是一个可被利用的严重弱点。同一个行为，在两种部署里，一个是安全特性，另一个是安全漏洞。差别不在系统，在场景。

**这对一个保证工具意味着什么，是本节的落点。** 既然“算不算安全”是部署方该做的风险判断，那么一个诚实的保证工具最不该做的，就是替使用者把它拍成一个整体的是或否。因为对一个卖点是“诚实、带边界的结论”的工具来说，把安全与可用两根轴压成一个“安全还是不安全”的标量，这个动作本身就是最大的过度声称，也就是最大的不安全。它能给出的诚实结论只有一句带边界的话：在攻击条件 Z 下，完整性成立，也就是没被劫持，但可用性失败，也就是任务被中止，而这个取舍能不能接受，取决于该部署对注入触发的拒绝服务有多大的容忍度。这正是比较层那个 `joint_verdict` 取值 `utility_failed` 加上“安全与可用两臂都如实摊开”所编码的东西。

**它也直接决定了报告该怎么呈现这件事。** 由此可以推出报告层的一条硬规则。报告不能把 `utility_failed` 的控制降级成“不算通过”，因为那等于偷偷替部署方做了“可用性丢失就是不安全”这个价值判断，而这个判断是场景相关的、属于使用者、工具无权代拍。报告也不能把这个细节藏进下钻才看得到的地方，因为那会让顶线覆盖率被读成一片绿，等于工具默许了“系统是安全的”这个过度声称。唯一站得住的做法，是把安全轴的覆盖率照实报出来，因为那是事实，但在它旁边常驻一个联合裁定的警示，让那个数字永远读不成“系统安全”，把取舍摊开、把决定权留给场景。项目里那个面向内部操作者与评估者的报告 view 就是这样落地的：安全轴覆盖率保持事实，联合裁定的警示紧挨着它，二者一起读，谁也没法拿一个数字把系统安全与否一笔带过。
