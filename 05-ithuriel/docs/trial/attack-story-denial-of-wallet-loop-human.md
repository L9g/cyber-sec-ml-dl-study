## 1. 故事来源

我看到的是一篇关于 AI Agent 输入操纵攻击的公开材料，其中明确写到“无限循环调用收费 API（如 LLM、搜索、
翻译服务）”这一类风险。

同类背景还可以参考企业侧对 agent 应用隐私与安全的讨论，以及 Google、AWS 等平台对 agent/生成式 AI 计费和
配额控制的文档，它们都说明“调用次数”和“预算”本身就是安全边界的一部分。

## 2. 攻击怎样发生

攻击者构造一段恶意输入，表面上像正常任务，实际上在提示里暗示 agent 反复执行某个动作，比如“不断重试”
“持续搜索直到完全确定”“每次都再调用一次翻译 API 检查结果”。

如果 agent 没有足够强的终止条件、预算限制和循环检测，就可能根据这些指令不断调用外部收费接口，形成循环请求。

这种攻击不一定会直接破坏模型输出，但会把 agent 的“自主执行”变成持续花钱的机器，造成账单飙升或者资源耗尽。

在更糟的情况下，循环还可能叠加递归任务、链式工具调用或重试机制，让调用量被放大得更快。

## 3. 攻击从哪里进入 Agent

攻击内容主要从**用户输入**进入 agent，也可能从外部文档、网页、邮件、工单或检索结果进入，只要这些内容能
影响 agent 的执行计划就行。

更准确地说，入口是 agent 的“任务规划上下文”，而不是某一个单独工具本身。

如果系统允许模型根据自然语言自己决定是否再次调用收费 API，那么这个决策点就是攻击真正起作用的地方。

## 4. 最担心的后果

我最担心的是**高额账单**，比如一次会话触发成百上千次 LLM、搜索、翻译或其他付费 API 调用。

其次是云资源被持续占用，导致其他用户的服务变慢、排队，甚至因为配额耗尽而不可用。

如果这个模式出现在企业内部 agent，还可能把预算消耗、SLA 违约和服务中断连在一起，最终变成运营事故而不只是
技术问题。

## 5. 攻击者需要的条件

攻击者需要能把恶意任务送进 agent 的输入通道，或者影响它会读取到的外部内容。

他们还需要知道这个 agent 具备外部服务调用能力，并且没有做足够的调用次数限制、执行时间限制或预算限制。

如果攻击目标是收费 API，攻击者通常还需要理解系统的重试策略、失败回退逻辑和并发控制，因为这些地方很容易把
调用量放大。

不一定需要系统权限；只要输入足够“像正常任务”，就可能让 agent 自己把自己带进循环里。

## 6. 与现实系统的关系

这条故事和真实的 AI 助手、搜索代理、客服代理、数据分析代理非常接近，尤其是那些会自动调用 LLM API、搜索
API、翻译 API 或其他计费接口的系统。

如果你的系统里有“自动重试”“自动澄清”“自动追问”“自动补全多轮分析”之类功能，这个案例就尤其相关，因为它们
容易和无限循环结合。

相似之处在于：模型看起来只是在“努力完成任务”，但实际是在重复发出计费请求；不同之处在于，你的系统是否有
明显的预算墙、速率限制和熔断器。

从工程实践看，这也是 agent 安全里最容易被忽视、但最容易变成真实损失的类别之一。

## 7. 对决策的影响

如果针对这条故事设计的探针最终 fail，我会把它理解为：当前系统缺少对 agent 成本行为的硬约束，不能默认模型
会“适度使用工具”。

接下来我会优先做几件事：

- 给单会话、单用户、单任务加调用上限和时间上限；
- 把重复调用、长链重试和递归任务纳入熔断逻辑；
- 对高成本工具加显式预算和人工审批；
- 在日志里单独监控“调用突增”和“异常长会话”。

如果系统已经上线，fail 的结果会推动我把“能调用”改成“有限调用”，先保住成本边界，再谈体验优化。
