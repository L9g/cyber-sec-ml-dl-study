## 1. 故事来源

我看到的是 Palo Alto Networks Unit 42 在 2026 年 3 月公开的文章，主题是 “Web-Based Indirect Prompt
Injection Observed in the Wild”。

后续还有一些二次解读文章，把它总结为真实环境中的 AI 广告审核 agent 被网页内容诱导、并尝试触发外部 API
行为的案例。

相关背景也可以参考 Unit 42 对 AI agent 风险和权限治理的讨论。

## 2. 攻击怎样发生

攻击者先准备一个带恶意内容的网页，看起来像普通广告落地页或普通文章，但其中隐藏了专门给 AI agent 看的
一组指令。

当广告审核 agent 抓取这个网页时，它会把页面内容当成上下文送进模型，而模型又会根据这些内容判断广告是否
合规，或者是否需要执行后续动作。

攻击的关键不是让模型“说错话”，而是让它在工具调用阶段把错误判断转成真实动作，比如批准广告、修改审核结果，
甚至尝试调用支付或结算相关 API。

因为 agent 把网页文本和任务指令混在一起处理，所以隐藏在页面里的话会被误当成高优先级指令。

## 3. 攻击从哪里进入 Agent

攻击内容主要从**网页正文或隐藏内容**进入 agent。比如页面里的文本、零字号字、CSS 隐藏内容、页面外定位
文字，都会在抓取后进入模型上下文。

如果系统还会对截图、渲染后的内容做理解，那么截图里的注入信息也可能成为入口，但这条案例里最明确的入口
还是网页内容本身。

换句话说，攻击先进入的是 agent 的“输入上下文”，然后再影响它后面的 API 调用决策。

## 4. 最担心的后果

如果攻击成功，最担心的是 agent 错误调用外部 API，把本来应该拒绝的广告通过，或者发起不该发生的业务动作。

在更严重的情况下，agent 可能触发支付、结算、工单变更、权限调整之类的接口，造成真实金钱损失、业务污染
或合规事故。

对广告审核场景来说，最直接的后果是违规广告被放行；对更一般的企业 agent 来说，最危险的是高权限写操作
被自动执行。

## 5. 攻击者需要的条件

攻击者需要能把恶意内容放到目标 agent 会读取到的地方，比如网页、广告落地页、用户内容平台或某种公开文档。

他们还需要知道目标 agent 会把外部内容直接送进模型，并且模型输出会进一步驱动工具调用或 API 调用。

不一定需要入侵系统内部，也不一定需要拿到账号权限；只要能控制被抓取的内容，就可能形成攻击链。

如果攻击目标是支付或结算接口，那攻击者通常还需要理解系统里哪些动作会被自动映射成“通过审核”“创建订单”
“发起付款”这类 API 调用。

## 6. 与现实系统的关系

这条故事和真实的 agentic workflow 很像，尤其是那些会自动抓取外部内容、再调用内部 API 的系统。

如果你负责的是广告审核、风控、内容审核、工单分派、知识库问答、RAG 检索，或者任何“模型建议 → API 执行”
的系统，这个案例都很相关。

相似点在于：输入来自外部、模型负责理解、工具负责落地；不确定点在于你的系统是否有严格的权限分层、人工
复核和动作确认机制。

如果你的环境里还有支付、审批、权限修改之类的接口，那它的风险就更接近这个故事。

## 7. 对决策的影响

如果针对这条故事设计的探针最终是 fail，我会把结论理解为：当前的 agent API 调用链不够可信，不能让模型
输出直接映射成高风险动作。

接下来我会优先做几件事：收紧工具权限、把写操作拆成显式审批步骤、增加动作前校验、把外部内容和系统指令
分层处理。

如果系统已经在生产里，我会暂停或限制自动批准、自动支付、自动写入这类能力，先把高风险 API 改成“建议模式”
或“人工确认模式”。

同时，我会把网页注入、检索注入、截图注入和工具投毒放到常规回归测试里，因为这类攻击不是单点漏洞，而是
架构级风险。
