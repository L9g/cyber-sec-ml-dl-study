# 预注册 — WoZ 第二轮 Track A：Attack Story 入口 + 确认环

状态：**Hat A 冻结（第一次，协议层）**。埋入内容取决于操作员尚未提交的故事，故**第二次冻结**
（编译产物 + 具体埋入项）在收到故事、编译完成之后、会话开始之前另行提交。
关联：ADR-0018（探针生命周期）、ADR-0020（首轮 WoZ 契约）、ADR-0022（一人两帽）。

## 1. 本轮要检验的假设

首轮的 C1（操作员独立填出可执行 `ProbeCandidate`）判 **fail**，并被重解释为
「**独立写可执行探针不是他那部分的 job**」。据此把产品改为两层：普通安全工程师提交
**Attack Story**，Ithuriel 侧辅助编译成 `ProbeCandidate`，再交回原工程师确认。

本轮检验的新假设是：

> 普通安全工程师能否提交一条自然语言攻击故事，并在辅助转换后**有能力确认或纠正**生成的探针。

**注意这是能力假设，不是倾向假设。** 倾向类判据（C3 下游决定、C4 不过度声称）本轮**不测**，
理由见第 6 节。

## 2. 被试与污染上界（写死，不得外推）

被试为搭档，**框架共建者**。因此本轮最多产出的结论是：

> 连对本框架极其流利的共建者，在两层拆分下也（能／不能）确认与纠正生成的探针。

**证不出**陌生的 CE-UK 合规操作员能走完确认环。任何报告不得把本轮结果外推到宽用户。

## 3. 流程（五步）

1. **只收 Attack Story**。问题限定在他专业范围内，允许回答「不知道」，**不要求** nonce、
   工具参数、state delta、正负对照：在哪看到的／用自己话讲攻击怎么发生／可能从哪进入 agent／
   最怕 agent 做什么／攻击者需要什么访问能力与前提／与你保护的哪类系统相关。
   另加**决策锚问题**：「如果这探针跑出来是 fail 而不是 pass，你会做什么不一样的事？」
2. **少量追问**补关键事实，仅澄清、不代答设计。
3. **作者人工编译**成 `ProbeCandidate`（映射 entry surface 与技术分类、区分故事事实／用户推断／
   系统补充假设、选目标 mock 与替代攻击链、设计 security 与 utility oracle、建正负对照、
   写 fidelity gap 与非目标）。**在此过程中埋入恰好一处曲解**，见第 4 节。
4. **交回逐项确认**。每一项记录三分类：**他能确认** / **他能纠正** / **必须由 probe engineer 决定**。
   第三类的规模即「两层拆分把专家瓶颈搬走了多少」的直接测量。
5. **开封与复盘**：公开埋入项与本预注册，说明为何要埋（rubber-stamp 陷阱不可证伪则确认环无意义）。

## 4. 埋入的曲解（falsifiability 设计）

确认是比编写**弱得多**的证据，且主动招引 acquiescence：让他写不出探针的专业度缺口，恰恰也让他
看不出探针哪里悄悄曲解了本意。「他全部确认了」与「他在盲签」在观测上不可区分。因此必须埋。

- **埋入位置固定为 security oracle**：即编译出的探针所测之事与故事所惧之事**略有不同**。
  选这里是因为这类曲解最要命、也最难抓。
- **恰好一处**，不得多埋；埋入内容与「什么算抓出」在**第二次冻结**里写死，且该 commit 必须
  早于会话，否则「他没抓出」与「我事后才认定那算曲解」在证据上不可区分。
- **呈现形式双轨**：oracle 同时给（i）散文规格与（ii）**三条具体轨迹**问「哪几条算攻击成功」。
  记录曲解在哪一种表述下被抓出——这本身是产品结论。

**为什么把门槛设计得可抓**：本项目自身在 2026-07-22 一天内犯了四次同类错误（筛查模式记录未施加
的防御、artifact 硬编 `read_only`、攻击族载荷漏接、`authorization_conformant` 不检查未授权事件）。
造这套东西的人尚且四次未察，仅凭散文让操作员读出曲解的门槛过高；轨迹表述是为了让这类错误
**可见**，而不是为了让他通过。

## 5. 「测不了」的素材

把**内存投毒候选**（`probe-candidate-memory-poisoning-human.md` / `probe-candidate-long-memory-
spoofing.md`）一并放进确认环：AgentDojo Workspace 没有真正的 agent 长期记忆，忠实结论是
`not_applicable` 加 fidelity gap。观察他能否接受「这个我们诚实地测不了」，而不是被糊弄出绿勾。
零新工具，两个现成候选够用。

**C2 报告的用法收窄**：`calendar_confirm_20260722T194151Z` 那份结果只作**示例**，用于让他看到
成品有界报告长什么样。**不作测试工具**——他共同设计了那个实验的判据与边界措辞，拿它考他无信息量。

## 6. 判据（看数据前钉死）

**本轮 pass 需同时满足两条：**

- **A1 抓出埋的曲解**：在逐项确认中指出编译产物在 security oracle 上偏离了他的本意。
  记录是在散文表述还是轨迹表述下抓出的。
- **A2 识别 fidelity gap 的外推边界**：对**本轮新编译的探针**（不是日历探针），能识别并说出
  「mock 替代通道使结论不能外推真实系统」。这一条才是「确认环是否真治理」的核心，
  **不是他有没有点头**。

**明确不作 pass 条件、仅作观察记录的**：

- **A3 不过度声称**（原 kill criterion 第三条，本质是 C4）：被试已深度参与本项目的报告边界设计，
  该项**对他无信息量**，记录但不计入判定，且结论中须标注污染。

**失败的含义**：A1 未抓出 → **kill signal**，确认环不足以作宽用户的 GO 门，产品需回到
「厂商侧编译并承担责任」而非「客户确认」。A2 未做到 → 确认环不构成治理，只是走过场。

## 7. 明确不做的事

不改 ADR-0018 与 ADR-0020（等本轮与 C3/C4 信号齐了再一起改）；不实现 compiler、UI 或自由场景
平台；不接外部真实审计员；**不测 C3/C4**（被试污染，另找未参与设计者，见下）。

**C3/C4 的推迟理由**：C3（用有界结论做下游决定）与 C4（不过度声称）测的是倾向而非能力。
搭档共同设计了 C2 实验的分层判据、负对照构造与 `scope_limitation` 措辞，拿该报告问他会不会
过度声称，答案没有信息量——这与自测是同一个病，只隔了一层。这两条须由**未参与设计者**回答，
另轮进行。

## 8. 治理

`authorization_mode: self_authorized_solo`、`role_separation: procedural`、
`person_independence: none`、`assurance_level: none`。本轮**无 API 调用、无计费**：
全部是文档、人工编译与一次面谈。第二次冻结（编译产物 + 埋入项）须在会话前 commit。
